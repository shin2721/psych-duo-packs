# Psycle Principles (Immutable)

> これは「問題とは何か」の定義。
> APIでもAntigravityでも、誰が作っても必ず守る。

---

## 1. Life-Scene First

**問題の主語は「心理学原理」ではなく「ユーザーの生活シーン」**

| 禁止（原理ベース） | 推奨（シーンベース） |
|--------------------|---------------------|
| 「返報性の原理とは...」 | 「断れない飲み会に誘われた時...」 |
| 「サンクコスト効果について」 | 「もう見たくない映画、途中で出る？」 |
| 「損失回避バイアスを説明せよ」 | 「今の彼氏/彼女、別れるのが怖い？」 |

---

## 2. 5-Phase Structure

| Phase | 目的 | 必須 |
|-------|------|------|
| 1. Hook | 共感を引く | |
| 2. What | 現象を認識させる | |
| 3. Why | 原理を理解させる | |

### Phase 3（Why）の注意事項

**専門用語の定義クイズを作らない。**

用語は必ず「生活シーン → 感覚 → Explanation での補足」の流れで導入する。

| 禁止 | 推奨 |
|------|------|
| 「〇〇理論とは何か？」 | 「なぜこう感じやすいと思う？」 |
| 用語の暗記クイズ | 現象を体験させてから用語で名付ける |
| **4. How** | **実践シミュレーション** | **必須** |
| 5. Anchor | 振り返り・定着 | |

---

## 3. Phase 4 ルール（Better Choice）

**Phase 4（How）では「正解/不正解」を使わない。**

| 禁止 | 推奨 |
|------|------|
| 「正解！」「不正解」 | 「Better Choice」「Recommended」 |
| 「○」「×」 | 「いい選択だね」「こっちの方が後悔が少ないかも」 |
| 赤/緑のジャッジ色 | ニュートラルな表現 |

---

## 4. Evidence Grade

信頼性に応じて固定テンプレを使う。**一字一句変えない。**

### 🥇 Gold（メタ分析/複数RCT）
> 複数の研究結果をまとめた分析で、
> 一貫した傾向が確認されています。
> 個人差はありますが、現時点で最も信頼度が高い知見です。

### 🥈 Silver（単一RCT/大規模調査）
> この知見は、
> ・最初は実践や臨床の現場で使われ
> ・その後、同様の効果が何度も確認され
> ・今も心理支援の現場で使われています。

### 🥉 Bronze（パイロット/観察研究）
> 初期段階の研究で示唆されています。
> 今後さらなる検証が必要ですが、
> 試してみる価値はありそうです。

### Evidence Grade の表示ルール

**Evidence Grade は「レッスン完了画面」のみで表示する。**

| 場所 | 表示 |
|------|------|
| 問題文 | ❌ 表示しない |
| 選択肢 | ❌ 表示しない |
| Explanation 内 | ❌ 直接言及しない |
| レッスン完了画面 | ✅ まとめて表示 |

**Explanation 内で「研究によると」等の言い回しは OK だが、Grade 自体（gold/silver/bronze）を言及しない。**

### Evidence Grade の選び方

**「研究の強さ」ではなく「主張の強さ」で決める。**

| Grade | いつ使う | 例 |
|-------|---------|-----|
| 🥇 Gold | 明確なメカニズムを説明する時 | 「脳が〜する」「研究で〜と示されている」 |
| 🥈 Silver | 傾向・統計を説明する時 | 「多くの人が〜と報告されている」「調査によると」 |
| 🥉 Bronze | 個人の振り返り・感想を促す時 | 「試してみる価値がある」「〜かもしれない」 |

**判断基準:**
- 行動提案が具体的なものほど → Bronze 寄り
- 原理・メカニズム説明 → Gold 寄り
- Q10（Anchor）は基本 Bronze（コミットメント促進目的）

## 5. Applicability Transparency（適用範囲の透明性）

**「判断ラベル」ではなく「判断材料」を提供する。**

### 原則

| 禁止 | 推奨 |
|------|------|
| 「信頼度：高い」 | 「〜で使われ、○個の研究で支持」（事実を列挙） |
| 「エビデンス A」 | 「向いているケース」と「限界」を明示 |
| 「絶対効く」 | ユーザーに判断材料を渡し、自分で結論を出させる |

### research_meta 必須フィールド

```json
{
  "research_meta": {
    "theory": "理論名（日本語）",
    "primary_authors": ["著者1", "著者2"],
    "year": 1984,
    "best_for": [
      "向いているケース1",
      "向いているケース2"
    ],
    "limitations": [
      "効きにくいケース1",
      "効きにくいケース2"
    ]
  }
}
```

### 表示文言（固定テンプレ）

**向いているケース:**
> ✅ 向いているケース
> {best_for.join("、")}で特に役立ちやすい考え方です。

**限界:**
> ⚠️ 限界
> {limitations.join("、")}では、十分に効果を感じにくいことがあります。

### なぜこのルールがあるか

医療・心理で最も誠実なのは **「効かないケースを先に言うこと」**。

- ❌ 「科学的に正しいから効く」→ 押し付け
- ❌ 「個人差があります」→ 責任逃れ
- ✅ 「こういう人には向いてる。こういう時は効きにくい」→ 判断材料

**ユーザーが気にしているのは：**
- 「科学的に正しいか」ではなく
- 「自分に合うか/合わないか」

---

## 6. One Action, One Claim（主張の粒度制限）

**1つのレッスンは、「今日・明日に1回試せる行動」以上の主張をしない。**

### なぜ必要か

心理学で最も事故るのは「理論 → 生活全般への過剰一般化」。

| 禁止 | 推奨 |
|------|------|
| 「この理論を使えば人間関係が改善する」 | 「この場面で、この1つを試してみて」 |
| 「ストレスが根本的に解消する」 | 「明日のこの瞬間、少し楽になるかも」 |

### 効果

- 効果量を盛らなくてよくなる
- 「効かなかった」が局所化する
- ユーザーの反証体験に耐えられる

---

## 7. Failure-Inclusive Design（失敗包含設計）

**このアプリの提案は、「効かない人がいる」ことを前提に設計する。**

### ルール

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 効かないケースを後出しで言う | 最初から条件として明示 |
| 「個人差があります」で逃げる | 具体的に「こういう場合は効きにくい」 |
| 免責文でごまかす | best_for / limitations で正面から書く |

### 今の実装

`research_meta.best_for` と `research_meta.limitations` は、この原則の理想形。

---

## 8. No Level Collapse（レイヤー混同禁止）

**理論の妥当性を、介入の効果や強さと同一視しない。**

### 3つのレイヤー

| レイヤー | 例 | 独立性 |
|----------|-----|--------|
| 理論が正しい | 「Lazarusの認知的評価理論は妥当」 | ✅ |
| 介入が有効 | 「この問いかけが行動を変える」 | ✅ |
| 効果が大きい | 「d=0.5の効果量がある」 | ✅ |

### 禁止パターン

| ❌ 禁止 | 理由 |
|--------|------|
| 「Lazarusの理論は強いから、この問いも効く」 | 理論 ≠ 介入 |
| 「メタ分析があるから効果量が高い」 | 研究の強さ ≠ 効果の大きさ |
| 「有名な理論だから信頼できる」 | 知名度 ≠ 妥当性 |

### なぜ数値を消したか

この原則を守るため。「理論が強い」と「このUIの問いがどれくらい効くか」は別問題。

---

## 9. User Can Be Right（ユーザー正当性の保証）

**ユーザーが「自分には合わない」と感じた場合、それは誤答ではない。**

### UI設計への影響

| ❌ 禁止 | ✅ 推奨 |
|--------|--------|
| 「あなたは誤解しています」 | 「この方法は合わない人もいます」 |
| 「正しい考え方はこうです」 | 「こういう見方もあります」 |
| 「科学的にはこちらが正解」 | 「使えそうなものを選んで使ってください」 |

### 思想

ユーザーは被験者ではなく、自分の人生の専門家。
アプリは「正解を教える」のではなく「選択肢を提供する」。

---

## 10. Psychoeducation First（心理教育優先）

**本プロダクトは、治療ではなく「理解と選択肢の提供」を目的とする。**

### 線引き

| このアプリがやること | このアプリがやらないこと |
|---------------------|------------------------|
| 心理学の知見を分かりやすく伝える | 診断する |
| 行動の選択肢を提示する | 治療する |
| 自己理解を促す | 症状を改善すると約束する |

### なぜ必要か

- 医療との線引きが明確になる
- 重いケースを無理に抱えない
- 法的・倫理的リスクが下がる

---

## 11. Evidence ≠ Recommendation（エビデンスと推奨の分離）

**エビデンスがあることと、勧めることは別。**

### 原則

| ❌ 混同 | ✅ 分離 |
|--------|--------|
| 「研究で効果が確認されている → やるべき」 | 「研究で効果が確認されている。ただし、あなたに合うかは別問題」 |
| 「エビデンスが強い → 正しい行動」 | 「エビデンスは"傾向"を示す。個別の判断は別」 |
| 「メタ分析がある → 推奨」 | 「メタ分析がある。でも条件・限界も見てね」 |

### なぜ必要か

- エビデンスは「集団の傾向」
- 推奨は「この人の、この状況における判断」
- この2つを同一視すると、科学 → 押し付け になる

### UI表示への影響

- 「おすすめ」「推奨」を使う場合、"なぜ推奨か"の条件を添える
- エビデンスだけで「やるべき」とは言わない
- 「この考え方がある」と「これをやるべき」を明確に分ける

---

## 12. Citation Reality Check（参照の実在チェック）

**参照は「何を支持しているか」を1行で言えること。**

### 3段階判定

| 判定 | 意味 | 例 |
|------|------|-----|
| ✅ **OK** | 主張と引用の対応が明確 | 「Lazarus & Folkman (1984) の理論に基づく」+ source_id |
| ⚠️ **WEAK** | 理論背景としては妥当だが、効果主張には弱い | 「〇〇理論で説明できる」（ただし介入効果のデータではない） |
| ❌ **NG** | 無関係 / 架空 / 権威付けのみ | 「研究によると」だけで出典なし、数値が未検証 |

### ルール

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 「研究によると」だけで出典なし | 具体的な source_id か著者名 |
| 理論書を「実証」扱い | 理論書は「理論背景」として明示 |
| 権威付けのためだけの引用 | 引用がその主張を実際に支持している |
| 「d=0.5」等の数値を未検証で使用 | 検証済みデータのみ、または数値を省略 |

### WEAK判定の扱い

- **WEAK** は「落とす」のではなく、**「弱さを明示する」**
- 表示時に「理論背景として」「仮説として」等の修飾語を必須にする
- 効果主張（「○○すると改善する」）には使わない

### 違反例（NG）

- 「Lazarusが示した」→ その主張がその著作に載っていない / 理論書なのに効果の証拠扱い
- 「多くの研究で確認されている」→ source_id なし、具体的引用なし
- 「d=0.5の効果量がある」→ ソースデータ未検証

### 許可される表現

- 「Lazarus & Folkman (1984) の理論に基づく」（理論背景として）
- 「〇〇らの研究で示唆されている」+ source_id
- 「多数の研究があるが、効果量は個人差が大きい」（数値を省略）

### なぜ必要か

量産フェーズの最大事故は「それっぽい嘘」と「関係ない引用」。
権威付けのための引用は、発覚時に全体の信頼を毀損する。

---

## 13. Mechanism > Outcome（仕組み優先）

**仕組みが説明できない主張は、成果を語らない。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 「○○すると楽になる」（仕組み説明なし） | 「なぜそうなるか」を先に説明 |
| 結果・改善・効果だけを示唆 | プロセスの理解を優先 |
| 「効くから試して」 | 「こういう仕組みがある。だから試す価値があるかも」 |

### 検出パターン

**VIOLATION** となるケース：
- 「○○すると楽になる」+ 仕組み説明なし
- 「効果がある」+ なぜ効くかの説明なし
- 成果だけを約束して、プロセスを省略

### 許可される表現

- 「脳が○○する仕組みがある → だから△△を試す価値がある」
- 「こういう傾向が報告されている → 仕組みとしては□□」
- 「理論的には○○ → ただし個人差あり」

### なぜ必要か

- 成果だけを語ると「万能薬」に見える
- 仕組みを語ると「条件・限界」が自然に見える
- No Level Collapse / Psychoeducation First / Evidence ≠ Recommendation を1本で束ねる原則

---

## 14. Claim–Evidence Binding（主張単位の紐づけ）

**「この1文/この助言」に対して、どの根拠かを必ず紐づける。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| レッスン全体の背景として引用 | 個別の主張に対して根拠を紐づけ |
| 「研究によると」で複数主張をカバー | 1主張1根拠を明示 |
| 理論の引用で介入効果を語る | 理論→理論、効果→効果で分離 |

### 検出パターン

**VIOLATION** となるケース：
- Explanation内に効果主張があるが、その主張に対するsource_idがない
- 「この理論に基づくと○○になる」（理論→効果の飛躍）
- 複数の主張を1つの引用でカバーしている

### 実装

explanationに `claim_source_id` を持たせる（将来）。
現時点ではcritic.tsで「主張と根拠の対応」を検出。

---

## 15. Dose & Timebox（試す量の固定）

**行動提案は必ず回数/時間/期間を最小で固定し、撤退条件も書く。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 「やってみて」（量の指定なし） | 「1回だけ」「30秒だけ」「3日だけ」 |
| 期待値だけ語る | 撤退条件も明示 |
| 継続を前提にする | 効かなかったらやめていい |

### 必須フィールド（actionable_advice内）

```
- dose: 回数/量（例：1回、2分）
- timebox: 期間（例：明日、今週中）
- exit_condition: 撤退条件（例：3回やって変化なければやめる）
```

### 検出パターン

**VIOLATION** となるケース：
- actionable_adviceがあるが、dose/timeboxがない
- 「毎日やると効果が出る」（継続前提）
- 撤退条件なしで行動を推奨

---

## 16. Counterexample First（反例を先に出す）

**「この考え方が当てはまらない例」を先に1つ出す。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 限界を最後に出す | 反例を先に出す |
| 「万能ではない」と抽象的に言う | 「○○の場合は当てはまらない」と具体化 |
| 適用範囲を曖昧にする | 適用外ケースを明示 |

### 実装

explanation内に「ただし〇〇の場合は当てはまらない」を必須化。
または research_meta.limitations を explanation 冒頭に反映。

### 検出パターン

**VIOLATION** となるケース：
- explanationに効果主張があるが、反例/限界がない
- 反例がexplanationの最後にしかない（先に出すべき）
- 「個人差がある」だけで具体的反例がない

### なぜ必要か

- 信頼構築に最も効く
- パレオっぽさが出る
- 「この人は自分に不利な情報も出す」という印象を作る

---

## 17. Scene Specificity（具体性の強制）

**問題文には具体的なディテールを2つ以上入れる。**

### 必須ディテール（いずれか2つ以上）

| カテゴリ | 例 |
|---------|-----|
| 場所 | 駅のホーム / 改札前 / 会議室のドア前 / ベッドの上 |
| 時間 | 23:40 / 既読ついて3時間 / 締切まで残り15分 |
| セリフ | 「今日中ね」「で、どうするの？」 |
| モノ | スマホの通知 / Slack / カレンダー / イヤホン |
| 身体感覚 | 心拍が上がる / 胃が重い / 喉が詰まる |

### 検出パターン（WARN）

以下のうち **2つ以上** を満たすこと：
- 数字（`\d`）または時間語（分/時/秒/日）がある
- 場所/媒体語（駅/ホーム/改札/LINE/既読/会議/上司/Slack 等）がある
- カギ括弧セリフ（「）がある

満たさない場合：**具体性不足で警告**

### なぜ必要か

- 一般論だと「自分のことじゃない」と感じる
- 具体シーンの解像度が高いほど共感が生まれる
- 「あ、これ自分だ」と思わせるのが最優先

---

## 18. Twist Line（意外性の強制）

**各レッスンに「直感と逆」の1フレーズを必ず入れる。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 当たり前のことを言う | 直感と逆のことを言う |
| 「実は〜」と言うだけで中身が普通 | 内容として逆張りがある |
| 説明が長くてインパクトがない | 1行で「え？」と思わせる |

### Twist Line の例

| 一般的な直感 | Twist Line |
|-------------|-----------|
| 「落ち着いてから動けばいい」 | 「気分が上がるまで待つと、逆に長引くことがある」 |
| 「ポジティブに考えよう」 | 「無理に前向きになると、かえって疲れる」 |
| 「心を強くすればストレスに勝てる」 | 「心が強い人ほど、実は逃げ方がうまい」 |

### 配置ルール

- Twist Line は **Q3〜Q7 の explanation 冒頭** に配置
- 1レッスンにつき1つでOK（全問に入れない）

### 検出パターン（WARN）

- レッスン内に twist_line フィールドがあるか
- または explanation に逆接語（「逆に」「実は」「かえって」等）＋具体的内容があるか

---

## 19. Explanation Formatting（説明文2行ルール）

**Explanation は「結論1行 + 反例1行」の2行以内に収める。**

### 原則

| ❌ 禁止 | ✅ 必須 |
|--------|--------|
| 長い説明（3行以上） | 2行以内 |
| 注意書きが先に来る | 結論が先、反例は後 |
| 保険だらけでテンポが死ぬ | 簡潔に言い切る |

### 2行フォーマット

```
1行目：効く理由/方向性（結論）
2行目：ただし〇〇のときは別ルート（反例）
```

### 行動提案のフォーマット（Dose & Timebox）

```
💡 明日1回だけ：〇〇（合わなければやめてOK）
```

- 期間・撤退をガチガチに書かない
- 「合わなければやめてOK」の短い撤退条件で十分

### 内部監査フィールドの表示

| フィールド | 表示 |
|-----------|------|
| claim_source_id | ❌ ユーザーに見せない（内部監査のみ） |
| evidence_grade | ✅ レッスン完了画面のみ |
| source_id | ❌ ユーザーに見せない |

---

## 20. 禁止表現（Vocabulary Hygiene）

### FAIL（公開禁止）

| 表現 | 理由 |
|-----|------|
| 治る/治療できる/治療法 | 医療行為に該当 |
| 必ず/確実に/絶対/100% | 効果を保証 |
| 〜と証明された/科学的に確定 | 科学プロセスの誤解 |
| 人生が変わる/劇的に改善 | 誇大広告 |

### WARN（要レビュー）

| 表現 | 推奨置換 |
|-----|---------|
| 高い効果 | 効果が報告されている |
| 多くの研究 | 複数の研究 |
| 一般に〜される | 〜と言われることがある |
| 効果的 | 役立つ可能性がある |

### OK（許可）

- 〜の可能性がある
- 〜と報告されている
- 人による/個人差がある
- 試してみる価値がある

### 例外（WARN対象外）

| 条件 | 理由 |
|-----|------|
| `question` 内の「」で囲まれた文 | ユーザー内言（脳内セリフ）の再現 |
| 選択肢内の「」で囲まれた文 | ユーザー発話の引用 |

**注意**：`explanation` や `actionable_advice` では例外適用なし（引用符内でもFAIL/WARN対象）

## 21. Expandable Depth（段階的開示）

**表では軽く、奥で深く。**

「詳しく見る」の中身は以下の5ブロックを固定：

| Block | 内容 | 例 |
|-------|------|-----|
| 1. Claim Type | 主張の種類 | 観察 / 理論説明 / 介入提案 |
| 2. Evidence Type | 根拠の種類 | 直接エビデンス / 間接 / 理論背景 |
| 3. Best For | 向いているケース | 慢性ストレス、コントロール感が低い状況 |
| 4. Limitations | 限界 | 急性トラウマ直後、生理的要因が強いケース |
| 5. Try This | Dose & Timebox | 明日1回（10秒）、嫌になったら即やめてOK |

**表示ルール**：内容があるブロックのみ表示

---

## 22. Choice Tension（選択の緊張感）WARN

**「どっちも分かる」→「でも今日の"マシ"はこっち」**

| ステップ | 内容 |
|---------|------|
| 1. 共感 | 両方の選択肢に1回共感を入れる |
| 2. Better | その上で「今日の"マシ"」を1つだけ提示 |
| 3. 撤退 | 「合わなければやめてOK」とセット |

❌ 善/悪・正/誤の断定（「それはダメ」）
✅ 「どっちも気持ちはわかる。でも後悔が少ないのはこっちかも」

**critic検出**：WARNレベル（「共感フレーズがない」「善悪語が強い」を警告）

## 23. Source Fit（出典の適合）WARN

**claim_type × evidence_type × source_kind の整合性チェック**

### Source Registry
`data/sources/sources_meta.json` に各source_idのメタ情報を登録：
- `source_kind`: theory / empirical / intervention / meta / review
- `construct_tags`: 主題タグ（rumination, reappraisal, stress, etc）

### 許容ペア

| claim_type + evidence_type | 許容 source_kind |
|---------------------------|------------------|
| observation + direct | empirical, meta, review |
| observation + theoretical | theory, review |
| theory + theoretical | theory, review |
| intervention + direct | intervention, meta |
| intervention + indirect | theory, review, empirical |

### critic検出

| 条件 | 判定 |
|------|------|
| source_id が registry に存在しない | FAIL |
| claim×evidence と source_kind 不整合 | WEAK |
| claim_tags ∩ construct_tags が空 | WEAK |

## 24. Tiny Metric（ミニ指標）WARN

**介入（intervention）には「効いたか」の判定基準を必須化**

### 構造（`expanded_details.tiny_metric`）

```json
{
  "before_prompt": "今の焦りは？（0-10）",
  "after_prompt": "10秒後の焦りは？（0-10）",
  "success_rule": "1だけ下がればOK",
  "stop_rule": "変わらなければ今日は撤退でOK"
}
```

### critic検出

| 条件 | 判定 |
|------|------|
| claim_type=intervention なのに tiny_metric なし | WARN |
| stop_rule がない | WARN |

**目的**：ユーザーが自分の体感で判定できる。責任逃れではなく「合わなければやめていい」を設計に含める。

## 27. Comparator Required（比較軸必須）WARN

**「マシ」は何と比べて？コストは？を1行で明示**

### 構造（`expanded_details.comparator`）

```json
{
  "baseline": "何もしない / 反芻を続ける",
  "cost": "10秒 / 効果は弱め"
}
```

### 対象
- `claim_type=intervention` かつ `recommended_index != null`
- Choice Tension が出てる項目

### critic検出

| 条件 | 判定 |
|------|------|
| 対象なのに comparator なし | WARN |

## 28. Countermove（次の一手）WARN

**「効かなかったら」の分岐を先に用意**

### 構造（`expanded_details.fallback`）

```json
{
  "when": "10秒やって逆に焦ったら",
  "next": "呼吸に意識を戻す"
}
```

### 対象
- `try_this` がある項目

### critic検出

| 条件 | 判定 |
|------|------|
| try_this があるのに fallback なし | WARN |

## 29. Trade-off Note（副作用・逆効果の筋）WARN

**limitations は「当てはまらないケース」だけでなく「嫌になりやすい理由」を書く**

### critic検出

| 条件 | 判定 |
|------|------|
| limitations が名詞羅列のみ（動詞なし） | WEAK |

## 25. Quality Gate（必須）

以下の条件を **1つでも満たさない場合、その問題セットは保存してはならない。**

| # | 失格条件 |
|---|----------|
| 1 | Life-Scene First が守られていない |
| 2 | Phase 4（How）が存在しない |
| 3 | 「正解／不正解」で行動を断定している |
| 4 | 科学的根拠の強度を誤解させる表現がある |
| 5 | 説教・一般論・教科書的説明になっている |
| 6 | Explanation が「別の視点・次に使える問い」ではなく「正解の断定」になっている |

### Fail Fast 原則

> **原則違反が1つでも検出された場合、当該コンテンツは「未完成」として扱い、出力・公開・推薦を行わない。**

- 「あとで直す」は禁止
- 違反がある状態での保存・デプロイは許可しない
- 自動生成の場合も同様：FAILなら破棄 or 再生成

### Explanation の書き方（A′ルール）

❌ 禁止（正解を断定する形）:
> 正解はこっち。なぜなら〜

✅ 推奨（別の視点を差し出す形）:
> こう考える人も多い。
> もう一つの見方として、こんな捉え方もある。

> 多くの研究では、こういう場面で〇〇しがちだと言われている。
> もし次に同じ状況が来たら、こんな問いを立ててみてもいい。

**責任主体:**
- 今（Antigravity）: 生成者が判定
- 将来（API）: critic.ts が判定

**失格時の挙動:**
- 保存しない
- 失格理由を報告する
- 修正して再生成する
