# Mastery Proof: Procrastination

## Summary

このドキュメントは、`先延ばし` という 1 テーマだけで

- Core
- Mastery
- Return
- Refresh

をどこまで重複感なく切れるかを検証するための proof である。

目的は `反芻` 以外の recurring pain でも、同じ供給構造が成立するかを確認すること。

---

## Theme

- `theme_id`: `procrastination`
- recurring pain:
  - 重要な作業ほど手をつける前に別のことを始めてしまう
  - 締切が近づくほど気まずくなって余計に開けなくなる
  - 先延ばし後に自己嫌悪して、さらに動けなくなる

---

## Core Unit

Core は `先延ばしが起きる流れを見抜き、戻り方を作る` 本編。

### Core 1. 気づく
- `lesson_job`: 先延ばしが始まっていることに気づく
- `target_shift`: 「まだ始めてないだけ」から「先延ばしが起きている」に変わる
- `takeaway_action`: いま避けている対象が何かを言語化する

### Core 2. 見分ける
- `lesson_job`: 休憩と回避を見分ける
- `target_shift`: 「休んでいる」から「避けているかも」に変わる
- `takeaway_action`: 回復のための停止か、不快回避かを見分ける

### Core 3. 理由を理解する
- `lesson_job`: 先延ばしが怠惰ではなく感情回避になりやすいと理解する
- `target_shift`: 「意思が弱い」から「不快回避が起きている」に変わる
- `takeaway_action`: 着手前の嫌さを 1 語で書く

### Core 4. 証拠を見る
- `lesson_job`: 先延ばしの連鎖を証拠で見る
- `target_shift`: 「自分はダメだ」から「この流れが起きている」に変わる
- `takeaway_action`: きっかけ / 回避行動 / 代償を分けて見る

### Core 5. 最小着手を選ぶ
- `lesson_job`: 先延ばしから抜ける最小行動を選ぶ
- `target_shift`: 「全部やるかゼロか」から「最小着手でいい」に変わる
- `takeaway_action`: 60秒で終わる開始行動を 1 つ選ぶ

### Core 6. 再発時に戻る
- `lesson_job`: 先延ばし後に戻る
- `target_shift`: 「もう遅い」から「ここから戻れる」に変わる
- `takeaway_action`: 再開時の最初の1分を決める

---

## Mastery Set

Mastery は `同テーマを別価値で返す` ための変奏。

### Mastery 1. 朝いちの先延ばし
- `lesson_job`: 朝いちに着手を避けるパターンを見抜く
- `difference`: `scene_change`

### Mastery 2. 夜の先延ばし
- `lesson_job`: 夜に気まずさで先送りする流れを扱う
- `difference`: `scene_change`

### Mastery 3. 仕事タスクの先延ばし
- `lesson_job`: 面倒より評価不安で止まる場面を扱う
- `difference`: `scene_change`

### Mastery 4. 勉強の先延ばし
- `lesson_job`: 分からなさを避けて着手できない場面を扱う
- `difference`: `scene_change`

### Mastery 5. “気分が乗ったらやる” を疑う
- `lesson_job`: 着手条件の思い込みを疑う
- `difference`: `judgment_change`

### Mastery 6. “完璧に始める” を崩す
- `lesson_job`: 完璧主義と先延ばしの接続を切る
- `difference`: `judgment_change`

### Mastery 7. 先延ばし再発の初期で戻る
- `lesson_job`: タスクを閉じた直後に戻る
- `difference`: `relapse_context`

### Mastery 8. 先延ばし後の自己嫌悪を減らす
- `lesson_job`: 二次ダメージで止まる流れを弱める
- `difference`: `relapse_context`

### Mastery 9. 60秒着手を現実タスクに移す
- `lesson_job`: 最小着手を仕事・家事・勉強に移す
- `difference`: `transfer_context`

### Mastery 10. 先延ばしと不安回避を切り分ける
- `lesson_job`: 先延ばしを単なる怠惰として扱わない
- `difference`: `transfer_context`

結論:
- `先延ばし` でも Mastery は **10本** までは十分切れる
- それ以上は薄い言い換えが増えやすいので、Refresh / Replay 側へ回した方がよい

---

## Return Set

Return は短い再入場。新知識を入れない。

### Return 1. いま避けているものを特定する
- `lesson_job`: 再開前に対象を見つける
- `question_count`: `3`

### Return 2. 60秒着手を選ぶ
- `lesson_job`: その場で戻る
- `question_count`: `4`

### Return 3. 自己嫌悪から切り離す
- `lesson_job`: 失敗直後に再開を可能にする
- `question_count`: `5`

---

## Refresh Candidates

Refresh は新研究や新 evidence で既存 lesson を更新する。

### Refresh A. explanation_update
- `parent`: Core 3
- 先延ばしと感情調整の説明を review ベースで更新する

### Refresh B. intervention_update
- `parent`: Core 5
- 実行意図や task chunking の負荷が低い使い方へ更新する

### Refresh C. scene_update
- `parent`: Mastery 3
- 仕事タスクでの気まずさや通知回避の scene を追加する

### Refresh D. evidence_grade_update
- `parent`: Core 2
- 休憩と回避の切り分けを単発研究から review ベースへ昇格する

---

## Capacity Estimate

- Core: `6`
- Mastery: `10`
- Return: `3`
- Refresh candidates: `4+`

1テーマあたりの実用供給量の初期見積もり:
- **固定本編**: `6`
- **重複感なく返せる変奏**: `8〜10`
- **短い再入場**: `2〜3`
- **研究更新で差し替え可能な枠**: `4以上`

合計すると、`先延ばし` だけで **20本前後の実用スロット** を持てる。

---

## Decision

- `1テーマ = Core 6` を基本にする
- `Mastery 8〜10` を目標レンジにする
- `Return 2〜3` を標準にする
- `Refresh` は lesson 数を増やすより、既存 slot の更新を優先する

この proof で確認できること:
- `反芻` だけでなく `先延ばし` でも同じ供給構造が成立する
- recurring pain が違っても、`Core + Mastery + Return + Refresh` で量を確保できる
- 量を支えるのは新テーマ量産ではなく、`theme ごとの Mastery 設計` である
